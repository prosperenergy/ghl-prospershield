## 1. Repository Purpose

`ghl-prospershield` is a **static public website** (`ghl.prospershield.io`) that publishes a read-only, redacted operational inventory of Prosper's GoHighLevel CRM, Make.com, n8n, Telnyx, and related connector infrastructure. It is deployed via Netlify + Cloudflare DNS and intended for internal operators, AI-assistant context injection, and LLM ingestion.

**It is NOT a backend service.** There is no server, no API, no database, and no runtime secrets in this repo. The site's value is as a navigable map; the build process that populates the map runs entirely on Craig's local Mac.

## 5. Security Findings

### 5.1 HIGH — Hardcoded Local Machine Paths in Committed Build Script

**File**: `scripts/build-ghl-live-map.mjs`  
**Lines**: 4, 5, 6, 7, 34, 45

```
Line 4:  const root    = "/Volumes/Samgsung T9/04_Fleet-Ops/ghl";
Line 7:  const wikiDir = "/Users/craigstratton/prosper-brain/wiki/sales";
Line 34: const envPath = "/Users/craigstratton/.config/prosper/prosper.env";
Line 45: const envPath = "/Users/craigstratton/.config/prosper/prosper.env";
```

The build script is non-portable and hardcoded to a specific external drive and user directory on Craig's personal Mac. It **cannot run in CI, on any other machine, or after a disk rename**.

Additionally, note the typo `Samgsung` (should be `Samsung`) — confirming this is a fragile personal device reference.

**Severity**: HIGH (operational continuity risk; any rebuild or onboarding of a second operator would fail silently or with a confusing crash)  
**Remediation**: Replace hardcoded paths with environment variables (`GHL_DATA_ROOT`, `GHL_WIKI_DIR`, `PROSPER_ENV_PATH`) with fallbacks. Add a `--help` or startup check that prints a clear error if required paths are missing.

> **Note**: A stored memory claimed this issue had been fixed with `PROSPER_ENV_PATH` support. That fix is **not present** in the currently committed code (lines 34 and 45 still hardcode the path). The memory has been downvoted.

### 5.2 MEDIUM — Employee PII Published on Public Website

**File**: `data/ghl-live-map.json`  
**Paths**: `$.prosperMain.users[]`, `$.prosperMain.phoneAssignments[]`, `$.phoneRisks[]`

The committed data file — served publicly at `https://ghl.prospershield.io/data/ghl-live-map.json` — contains:

- **47 employee full names** in `prosperMain.users` (e.g., "Abdourahamane Diallo", "Abidemi Mary Adebayo")
- **15 real US phone numbers** with linked employee names in `prosperMain.phoneAssignments` (e.g., `+14104987548` linked to "Brian Job")
- **One forwarding phone number** (`+13049371183`) repeated across 9 assignment rows
- **Specific employee names in phone risk entries** (e.g., "Andrew Brown, Daniel Tirri, Jeff Restel, and Scott show incomplete/failed A2P 10DLC")

This data is indexed by search engines, AI crawlers (`robots.txt` allows all), and the `llms.txt` / `ai-context.md` files actively invite LLM ingestion.

**Severity**: MEDIUM (privacy/compliance risk for employees whose names and direct-dial numbers are now indexed publicly)  
**Remediation**: Decide intentionality. If this is intentional for operator use, consider adding a `noindex` signal for the JSON data files specifically. If not intentional, replace full names with roles/IDs and redact or omit real phone numbers from the committed JSON.

### 5.3 MEDIUM — n8n Webhook Path Slugs Publicly Committed

**File**: `data/ghl-live-map.json`  
**Path**: `$.n8n.workflows[].webhookPaths[].path`

Internal n8n webhook path slugs are published:
- `ghl-outbound-call`
- `hanna-ai-sms`
- `hanna-ai-sms-poller`

While the n8n host URL is not published, these slugs narrow the attack surface significantly if the host is ever enumerated or disclosed. An adversary knowing these paths could attempt replay attacks, probing, or flood the endpoints.

**Severity**: MEDIUM  
**Remediation**: Remove or generalize webhook path slugs from the public data. The schema can retain the workflow name and integration list without exposing the path slug.

---

### 5.4 MEDIUM — XSS Risk via Unsanitized innerHTML

**File**: `app.js`  
**Lines**: 43, 59, 91, 109, 128, 153, 158, 233, 307 (and others)

`app.js` calls `innerHTML` in ~12 locations to render content sourced directly from `fetch("./data/ghl-live-map.json")`. For example:

```js
// app.js:109
$("attention-list").innerHTML = rows
  .map((item) => `<button ... ><strong>${item.title}</strong><p>${item.detail}</p></button>`)
  .join("");
```

If the JSON data file were ever compromised (e.g., via a supply-chain commit, a MITM on the CDN, or a misconfigured Netlify deploy key), malicious HTML/script payloads in `title` or `detail` fields would execute in any visitor's browser.

There is **no Content-Security-Policy** in `index.html` and no `netlify.toml` setting security headers.

**Severity**: MEDIUM (exploitable only if the data source is compromised, but no defense-in-depth exists)  
**Remediation**: Add a `netlify.toml` with a `Content-Security-Policy` header (`default-src 'self'; script-src 'self'; object-src 'none'`) and `X-Frame-Options: DENY`. For high-value fields rendered via `innerHTML`, consider escaping HTML entities before interpolation, or switch to `textContent` + DOM construction for those fields.

---

### 5.5 LOW — No CSP or Security Headers

**Files**: `index.html`, (no `netlify.toml` present)

The published site sets no security headers. Browsers receive no guidance on:
- Script execution policy (no CSP)
- Framing protection (no `X-Frame-Options`)
- MIME sniffing (no `X-Content-Type-Options`)
- Referrer leakage (no `Referrer-Policy`)

**Severity**: LOW  
**Remediation**: Add a `netlify.toml` with a `[[headers]]` block. This is a one-file additive change with no risk.

---

### 5.6 LOW — MAKE_API_TOKEN_OP Stale Token Referenced as Actionable

**File**: `data/ghl-live-map.json`  
**Path**: `$.credentialMap.systems[1].loginStatus`

The JSON explicitly notes: *"MAKE_API_TOKEN_OP previously returned 401 and should be treated as stale or vault-only until reauthorized."* This is a public declaration of a known broken credential. While no value is exposed, advertising that a specific credential ref is stale and un-reauthorized can assist targeted social engineering.

**Severity**: LOW  
**Remediation**: Generalize to "some tokens in vault need revalidation" without naming the specific ref in the public map.

## 7. Top 5 Prioritized Recommendations

### R1 — [HIGH] Fix Hardcoded Local Paths in Build Script

Replace the four hardcoded personal Mac paths in `scripts/build-ghl-live-map.mjs` (lines 4, 7, 34, 45) with environment variables (`GHL_DATA_ROOT`, `GHL_WIKI_DIR`, `PROSPER_ENV_PATH`). Add a startup guard that exits with a clear error message if `GHL_DATA_ROOT` is unset. This unblocks any second operator and any future CI automation.

**Effort**: Small (< 1 hour). **Risk**: None — additive change; the script currently only runs on Craig's Mac anyway.

---

### R2 — [MEDIUM] Audit and Scope PII in the Public Data File

Review `data/ghl-live-map.json` and decide whether publishing 47 employee full names and 15 real phone numbers publicly (indexed by Google and LLM crawlers) is intentional and has staff consent. Consider replacing personal names with role identifiers (e.g., "Solar Consultant 1") and redacting or omitting direct-dial numbers from the committed file. The operational map can still be fully functional with role-based labels.

**Effort**: Medium (data restructure + rebuild). **Risk**: Low — the published site would show role labels instead of names, which is safer and still useful.

---

### R3 — [MEDIUM] Add `netlify.toml` with Security Headers

Create `netlify.toml` at the repo root with `Content-Security-Policy`, `X-Frame-Options: DENY`, `X-Content-Type-Options: nosniff`, and `Referrer-Policy: strict-origin-when-cross-origin`. This is an additive one-file change that immediately hardens the public site.

**Effort**: Trivial (< 30 minutes). **Risk**: Very low — if the CSP is too strict it can be relaxed; start with `default-src 'self'` since the site has no external scripts.

---

### R4 — [MEDIUM] Remove n8n Webhook Path Slugs from Public Data

Remove the `path` field from `$.n8n.workflows[].webhookPaths[]` entries in `ghl-live-map.json`. The workflow name, integration list, and node type are sufficient for the public map's purpose. This is a one-time data edit; the `build-ghl-live-map.mjs` script should also be updated to omit path slugs when it next runs.

**Effort**: Small (data edit + script update). **Risk**: None — informational change only.

---

### R5 — [MEDIUM] Add a Minimal GitHub Actions CI Workflow

Add `.github/workflows/ci.yml` that runs on every push and PR to `main`. At minimum: (1) validate `data/ghl-live-map.json` and `data/ai-index.json` are valid JSON using `node -e "JSON.parse(require('fs').readFileSync(...))"`, and (2) run `node scripts/build-ai-index.mjs` to verify it completes without error. This catches JSON syntax errors and broken regeneration scripts before they deploy.

**Effort**: Small (< 1 hour). **Risk**: None — additive; does not touch deployment.
